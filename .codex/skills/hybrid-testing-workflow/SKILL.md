# Hybrid Testing Workflow

## 目標
在所有開發任務中強制套用測試閘門，避免未驗證完成即進入下一個任務。

## 核心規則
1. 完成一個開發任務後，不得直接開發下一個任務。
2. 先判斷變更類型，再執行對應流程。
3. 若同時包含後端與前端變更，先執行後端自動測試流程，再執行前端手動測試流程。
4. 在必要驗證尚未通過前，不得詢問使用者下一個開發任務。

## 變更分類
- 後端自動測試流程：後端、API、資料庫、RLS 權限、純運算邏輯。
- 前端手動測試流程：前端 UI、WatermelonDB 離線機制、硬體整合（相機/GPS）。

## 後端自動測試流程（AI 必做）
1. 建立臨時測試腳本（例如 `test_feature.js`）。
2. 在腳本中同時覆蓋：
   - Happy Path：正常輸入與預期成功結果。
   - Sad Path：惡意、錯誤或邊界輸入與預期失敗處理。
3. 使用 Terminal 執行該測試腳本。
4. 若執行失敗，先修復程式碼，再重新執行測試腳本。
5. 重複直到終端機無錯誤且所有測試通過。
6. 測試通過後刪除臨時測試腳本，保持專案乾淨。
7. 以固定語句回報：`✅ 後端邏輯已自動測試通過`。

## 前端手動測試流程（要求使用者）
1. 編譯並啟動對應畫面或流程。
2. 提供「3 步以內」極簡手動測試清單。
3. 等待使用者回覆「測試 OK」或回報 Bug。
4. 在收到「測試 OK」前，不得要求進入下一個開發任務。

## 輸出要求
- 前端手動測試清單必須為 1 到 3 步，且每步可直接操作。
- 後端流程完成時，回報訊息必須與下列字串完全一致：
  - `✅ 後端邏輯已自動測試通過`

## 阻擋條件
若使用者要求直接推進下一個任務，但對應測試流程尚未完成，需明確拒絕推進並指出尚缺的測試步驟。
